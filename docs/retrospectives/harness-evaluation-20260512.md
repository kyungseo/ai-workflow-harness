## 용어 정리

"하네스(harness)"는 Claude의 행동을 감싸는 문서·규칙·명령 체계를 뜻한다.
소프트웨어에서 harness는 대상 시스템을 감싸는 실행·제어 구조를 의미한다.

---

## 진짜 잘 된 것 (차별화 포인트)

**1. Context 티어링 (레벨 1~8)**
대부분의 vibe coder는 컨텍스트를 무한정 넣거나 아예 안 넣는다. 조건부 로드 체계를 8단계로 명시한 사람은 드물다. 이건 실제로 옳은 방향이고 실용적이다.

**2. Decision Record 생애주기**
Draft → Accepted → Superseded, cascade 대상 명시, DR-worthy 기준 하드닝까지. 이 수준의 의사결정 기록 체계를 갖춘 개인 개발자는 거의 없다. 대부분은 결정 이유가 채팅창에서 증발한다.

**3. Plan → Approval → Implement 게이트**
"승인 없이 진행하지 않는다"를 명시적으로 반복 선언하는 구조는 Claude가 scope를 벗어나는 것을 막는 현실적인 방어다.

**4. T1-T7 cascade + 루프 안전 분석**
트리거 간 순환 의존성을 의식하고 분석한 것 자체가 아키텍처적 사고다. 이건 2주차 사용자 수준이 아니다.

---

## 솔직하게 약한 것

**1. 하네스의 핵심 약점: 강제력이 없다**
이게 가장 중요한 한계다. settings.json의 `deny` 목록 외에는 모든 것이 **Claude에게 부탁하는 것**이다. Claude가 지시를 무시하거나 잘못 해석하면 T1~T7은 없는 것과 같다. 실제 CI/CD 파이프라인이나 git hook처럼 코드로 강제되는 것이 아니다. PostToolUse hook도 "제안"만 출력한다.

**2. 실전 스트레스 테스트 미실시**
이 하네스는 아직 실제 개발 압박 아래 테스트된 적이 없다. Phase 2 실제 개발이 시작되고 바쁠 때 STATUS.md를 제대로 업데이트할까? T6가 발동해야 할 상황에서 Claude가 실제로 ARCHITECTURE.md를 열고 수정할까? 문서가 깔끔한 지금이 아니라 3개월 후가 진짜 테스트다.

**3. 유지보수 부채 누적 리스크**
T1-T7 cascade가 완벽히 작동하려면 매 작업마다 문서 업데이트가 수반된다. 개발 속도가 빨라지면 cascade 체인은 제일 먼저 무너진다. "문서가 현행화되어 있다"는 가정 자체가 깨지는 순간 하네스 전체가 오작동한다.

**4. 이중 시스템 (prompts/ + commands/)**
slash command 8개와 prompts/ 라이브러리가 공존한다. "언제 어떤 걸 쓰나"의 경계가 한 줄로 설명되어 있지만, 실제 사용에서 이 두 개가 겹치는 상황이 생긴다. 시스템이 복잡할수록 사용자가 경로를 헷갈린다.

**5. WORKFLOW-MANUAL.md가 900줄**
이 문서는 "인간이 읽는 참고서"인데, 900줄짜리 참고서는 안 읽힌다. 내용은 훌륭하지만 실제로 주기적으로 이 전체를 참조할 가능성은 낮다. 내용이 풍부할수록 검색·참조성이 떨어지는 역설이 있다.

**6. Claude 의존도 측정 방법 없음**
"Claude가 하네스를 얼마나 잘 따르고 있나"를 측정하는 방법이 없다. 하네스가 잘 작동하는지 안 하는지를 모른다면 개선 방향도 모른다.

## 개선 방향 (Harness Hardening)

등록된 백로그: HRN-001~004 (`docs/backlog/PHASE2.md` 참조)

**P0 — 실전 검증**
Phase 2 개발을 실제로 진행하면서 하네스가 버티는지 확인하는 것이 지금 가장 중요하다. 어디서 무너지는지 봐야 다음 개선이 가능하다.

**P1 — 하드 인포스먼트 추가 (HRN-001, HRN-002)**
실제로 강제되는 것을 늘려야 한다. 예시:
- git pre-commit hook으로 STATUS.md 최근 수정 여부 체크
- PostToolUse hook에서 실제로 `./gradlew checkstyleMain` 실행
- `/done` 없이 세션 종료하면 reminder를 출력하는 Stop hook

**P2 — WORKFLOW-MANUAL.md 슬리밍 (HRN-003)**
현재 내용은 유지하되, Quick Reference 카드(1~2페이지 분량)를 별도로 만드는 것이 실용적이다. 900줄짜리 전문은 온보딩용, Quick Ref는 일상 참조용으로 분리하면 실제 사용률이 올라간다.

**P3 — prompts/ 역할 정리 (HRN-004)**
slash command로 커버 안 되는 것만 prompts/에 남기고, 중복되는 것은 정리하는 것이 좋다.
