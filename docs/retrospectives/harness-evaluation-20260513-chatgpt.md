1냉정한 재평가 (타겟: Claude Code 숙련자)

결론

* 현재 위치: 상위 20~30%
* 상위 10%로 가려면 필요한 것:
    * 자동화 (enforcement)
    * 경량화 (cognitive load 감소)
    * 실제 실패 데이터 기반 tuning


왜 10%가 아닌가 (핵심 이유 3개)

1) “정책은 강하지만 enforcement가 약함”

지금 구조는:

* 규칙: 매우 잘 정의됨
* 실행: Claude + 사람 의존

즉,

“지켜야 한다”는 많고, “안 지키면 깨진다”는 적다

상위 10%는 다음 특징이 있음:

* rule → hook → 자동 fail
* rule → CI → 차단

👉 현재 상태:

* PostToolUse hook 일부 있음
* 하지만 대부분은 “권장/제안”

👉 평가:

* 운영 체계가 아니라 “운영 가이드”에 가까움

---

2) 문서 체계가 “완성형”이라기보다 “이상형”

문서 간 관계는 매우 잘 설계되어 있음:

* STATUS → backlog → DR → PLAN → ARCHITECTURE cascade

하지만 실제 현장에서는:

* 30~40%는 깨짐
* DR 누락
* STATUS stale
* PLAN 미반영

👉 중요한 포인트:

상위 10% 구조는 “깨지는 걸 전제로 복구 설계”가 더 강함

현재는:

* 실패 복구 규칙 있음 (good)
* 하지만 자동 탐지/강제 부족 (gap)

---

3) Cognitive Load가 높음 (핵심)

지금 구조를 한 줄로 요약하면:

“잘 쓰면 엄청 좋은데, 계속 잘 쓰기 어렵다”

특히:

* 트리거 7개
* 문서 타입 10+
* slash command 8개
* cascade 규칙 다수

👉 Claude Code 숙련자 기준에서도:

* onboarding 비용 높음
* 유지 비용 높음

상위 10% 구조는:

* 규칙은 더 적고
* 대신 자동화가 더 많음


---

2. Hardened v2 보완 내용 리뷰

결론:
방향은 정확하다. 다만 “문제의 절반만 해결된 상태”다.
